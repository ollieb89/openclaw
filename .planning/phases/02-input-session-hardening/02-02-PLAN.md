**Step 1 — Create `src/security/session-access.ts`:**

Define and export:

```typescript
export type SessionAccessDecision = {
  allowed: boolean;
  reason?: string;
};

export type SessionAccessType = "transcript" | "memory" | "metadata" | "list";
```

Implement `authorizeSessionAccess(params: { callerSessionKey: string; targetSessionKey: string; accessType: SessionAccessType; config: OpenClawConfig }): SessionAccessDecision`:

**Access rules (in order of evaluation):**

1. **Same session** (`callerSessionKey === targetSessionKey`): Always allowed.

2. **Main/gateway session as caller**: If the caller is the main session (check via the config's `session.mainKey` or key equals `normalizeMainKey()`), always allowed. The main session is the administrative context and needs full visibility.

3. **Same agent, cross-session**: Extract agent IDs using `resolveAgentIdFromSessionKey()` from `src/routing/session-key.ts`.
   - `metadata` and `list` access types: Allowed (sessions within the same agent can see each other exists).
   - `transcript` and `memory` access types: **Denied**. Return `{ allowed: false, reason: "Cross-session transcript/memory access denied within same agent" }`.

4. **Cross-agent**: Defer entirely to the existing `createAgentToAgentPolicy(config)`. If A2A policy says allowed, allow. If denied, deny with reason "Agent-to-agent access denied by tools.agentToAgent policy".

**Important:** When a cross-session access is denied, emit a security event via `emitSecurityEvent()`:
```typescript
emitSecurityEvent({
  eventType: "policy.violation",
  timestamp: new Date().toISOString(),
  sessionKey: callerSessionKey,
  severity: "warn",
  action: "blocked",
  detail: `Cross-session ${accessType} access denied: ${callerSessionKey} → ${targetSessionKey}`,
  meta: { callerSessionKey, targetSessionKey, accessType },
});
```

**Step 1 — Wire into `sessions-history-tool.ts`:**

After the existing A2A policy check block (around line 219-237), add a same-agent cross-session check. Import `authorizeSessionAccess` from `../../security/session-access.js`. After the `resolvedKey` is determined and the existing sandbox/A2A checks pass, add:

```typescript
// Cross-session isolation (same agent, different sessions)
if (!isCrossAgent && requesterInternalKey && resolvedKey !== requesterInternalKey) {
  const accessDecision = authorizeSessionAccess({
    callerSessionKey: requesterInternalKey,
    targetSessionKey: resolvedKey,
    accessType: "transcript",
    config: cfg,
  });
  if (!accessDecision.allowed) {
    return jsonResult({
      status: "forbidden",
      error: accessDecision.reason ?? "Cross-session access denied",
    });
  }
}
```

This integrates cleanly AFTER the existing A2A check (which handles cross-agent) and BEFORE the data fetch (`callGateway("chat.history")`), enforcing isolation at the data access boundary.

**Step 3 — Wire into `memory-tool.ts`:**

The memory search tool scopes by `agentId` but doesn't enforce session-level isolation for session transcript chunks. Read the full file to understand the search result structure.

After search results are returned from `manager.search()`, filter results whose source is a session transcript: if the result's `path` or `source` indicates it came from a session transcript (not from `MEMORY.md` or `memory/*.md` files), check whether the caller session has access to that session's data using `authorizeSessionAccess` with `accessType: "memory"`.

**How to detect session transcript sources:** Memory search results include a `path` field. Session transcript paths contain `sessions/` in the path. Global memory files (`MEMORY.md`, `memory/`) do NOT — these are shared within the agent and should never be filtered.
