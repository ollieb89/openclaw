Read `src/auto-reply/reply/get-reply-run.ts` to understand the inbound message flow. This is the common path where all channel messages reach before agent processing.

At the earliest point after the message content and channel are available (but before the agent turn begins), add a call to `screenInput()`:

1. Import `screenInput` from `../../security/input-screening.js`
2. Extract the `InputDetectionConfig` from `loadConfig().security?.inputDetection ?? {}`
3. Call `screenInput({ content: messageContent, channel, sessionKey, config: inputDetectionConfig })`
4. If the action is `"block"`:
   - Do NOT pass the message to the agent
   - Return a brief rejection message: "Message blocked by security policy"
   - Log at debug level that the message was blocked
5. If the action is `"warn"`:
   - Prefix the message content with a security notice before passing to agent (similar to `wrapExternalContent` pattern): `"[Security notice: potential injection detected (score: X.XX)] "` â€” this tells the agent to be cautious but still process the message
6. If the action is `"log"` or `"allow"`:
   - Pass through normally (the security event was already emitted by `screenInput` for "log" actions)

**Important:** The `screenInput` call already handles security event emission internally. The pipeline integration only needs to handle the response action.
