**Create `src/security/events.ts`** with typed security event definitions:

```typescript
export type SecurityEventType =
  | "auth.attempt"
  | "auth.success"
  | "auth.failure"
  | "tool.call"
  | "tool.denied"
  | "injection.detected"
  | "policy.violation";

export type SecurityEventSeverity = "info" | "warn" | "critical";

export type SecurityEvent = {
  eventType: SecurityEventType;
  timestamp: string;          // ISO 8601
  sessionKey?: string;
  channel?: string;
  severity: SecurityEventSeverity;
  action: string;             // What was done: "allowed", "blocked", "logged"
  detail?: string;            // Human-readable context
  meta?: Record<string, unknown>;  // Additional structured data
};
```

Align severity names with the existing `SecurityAuditSeverity` in `src/security/audit.ts` (which uses the same `"info" | "warn" | "critical"` -- keep consistent).

**Create `src/security/event-logger.ts`** wrapping the existing SubsystemLogger:

```typescript
import { createSubsystemLogger } from "../logging/subsystem.js";
import type { SecurityEvent, SecurityEventType, SecurityEventSeverity } from "./events.js";

const securityLogger = createSubsystemLogger("security");

export function emitSecurityEvent(event: SecurityEvent): void {
  const { severity, eventType, ...meta } = event;
  const message = `[${eventType}] ${event.action}${event.detail ? `: ${event.detail}` : ""}`;

  // Route to appropriate log level based on severity
  if (severity === "critical") {
    securityLogger.error(message, meta);
  } else if (severity === "warn") {
    securityLogger.warn(message, meta);
  } else {
    securityLogger.info(message, meta);
  }
}
```

Important: The function must NEVER throw. If logging fails, it should fail silently (the SubsystemLogger already handles this via try/catch internally). Do NOT add any try/catch -- the underlying logger handles it.

Add `emitSecurityEvent()` calls at three integration points. Import from `../security/event-logger.js` (or appropriate relative path).

**1. `src/gateway/auth.ts` -- Auth attempts:**

Read the full file first to understand the auth flow. Find where `GatewayAuthResult` is returned (the function that resolves auth -- likely returns `{ ok: true/false, method, reason }`).

At every point where an auth result is determined:
- On success (`ok: true`): emit `{ eventType: "auth.success", severity: "info", action: "allowed", detail: method, meta: { method } }`
- On failure (`ok: false`): emit `{ eventType: "auth.failure", severity: "warn", action: "blocked", detail: reason, meta: { method, reason } }`
- On rate-limited: emit `{ eventType: "auth.failure", severity: "warn", action: "rate_limited", detail: "too many attempts", meta: { retryAfterMs } }`

Always include `timestamp: new Date().toISOString()`. Session key and channel may not be available at auth time -- omit if not available.

**2. `src/agents/tool-policy.ts` -- Tool allow/deny:**

Read the full file first. Find where tools are checked against the allow/deny policy (likely a function that returns whether a tool is allowed).

At the decision point:
- On deny: emit `{ eventType: "tool.denied", severity: "warn", action: "blocked", detail: toolName, meta: { toolName, profileId } }`
- On allow: emit `{ eventType: "tool.call", severity: "info", action: "allowed", detail: toolName, meta: { toolName, profileId } }`

Note: Only emit at the policy enforcement boundary, NOT on every individual tool invocation. If the file has a function like `isToolAllowed` or `resolveToolPolicy`, that's the right place.

If tool allow events would be too noisy (every tool call), only emit on DENY. Use your judgment based on the code structure -- if there's a clear policy check function, emit on both. If tools just flow through without an explicit check, emit only on deny.

**3. `src/security/external-content.ts` -- Injection detection:**

- If matches.length > 0: emit `{ eventType: "injection.detected", severity: "warn", action: "logged", detail: \`${matches.length} suspicious pattern(s) detected\`, meta: { patterns: matches } }`

Find the caller of `detectSuspiciousPatterns` in this same file (likely `wrapExternalContent` or similar) and add the event emission there, NOT inside `detectSuspiciousPatterns` itself (keep it a pure function).

For all three integration points:
- Import `emitSecurityEvent` from the security event-logger
- Import `SecurityEvent` type if needed for construction
- Use `new Date().toISOString()` for timestamp
- Never let event emission block the main flow
